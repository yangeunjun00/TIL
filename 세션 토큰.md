# 인증방식

사용자가 웹사이트에 로그인을 할 때 서버에서 사용자를 인증하는 방식에는 크게 **세션**기반 인증과 **토큰**기반 인증 이렇게 두가지가 있습니다.

오늘은 이 두 가지 인증방식에 대해 설명해보겠습니다.

그 전에 잠깐 설명해보자면

세션은 **사용자가 인증에 성공한 상태**를 말하고
토큰은 민증이나 학생증 과 같은 본인을 인증할 수 있는 증표이다.

또 세션은 서버에서 인증정보를 가지고 있는 형태이다.
그래서 서버에서 과도한 부하가 생길 수 있다.
이를 해결하기 위해 토큰이 등장한 것 이다.
토큰은 그 반대로 클라이언트에서 인증정보를 가지고 있는 형태이다.

## 세션기반 인증 방식

세션기반 인증 방식이 어떻게 이루어지는지 웹사이트에서 로그인할 때 이루어지는 과정에 대해 자세하게 설명해보겠다.

- 사용자가 ID/PW 로 웹사이트에 로그인 요청을 보낸다.
- DB에 있는 사용자인지 인증점검을 한다.
- DB에 있는 사용자면 세션 ID를 생성한 후 서버 메모리에 저장
- 클라이언트에 쿠키의 헤더에 세션 ID를 담아 전달
- 클라이언트에 세션 ID 저장
- 클라이언트에서는 앞으로 요청 할 때 마다 쿠키에 세션 ID를 담아서 요청
- 서버에서는 해당 세션 ID를 가진 세션 데이터를 조회
- 세션이 유효하면 요청에 해당하는 응답

### 장단점

앞서 말했듯이, **서버에서 인증 상태 관리를 해서 서버에 부담이 심해진다**

또 유저의 세션 데이터는 한 서버에만 저장되기 때문에 다른 서버에서도 사용한다면 서버 간 데이터 공유가 필요해진다

이러한 서버의 부담을 해결할 수는 없을까?
이에 대한 해결책으로 토큰이 등장하게 되었다.

## 토큰기반 인증 방식

토큰기반 로그인 과정은 다음과 같다.

- 사용자가 ID/PW 로 로그인 요청
- DB에서 유저 있는지 인증점검
- 유효하면 해당 유저에 대한 토큰 생성
- 클라이언트로 토큰 전송(Authorization 헤더)
- 클라이언트에서 토큰 저장
- 리소스 요청과 함께 토큰을 서버로 전달(Authorization 헤더)
- 유효한 토큰인지 검증(비밀키)
- 유효하면 요청에 해당하는 응답 전송

### 장단점

토큰을 이용하면 서버에서 유저의 인증 상태를 저장할 필요 없이 클라이언트에서 관리를 하므로 서버의 부담을 줄여줄 수 있다.

또한 하나의 토큰으로 다른 서버에도 인증을 할 수 있다.

다수의 서버에 공동 세션 데이터를 가지고 있을 필요가 없는 것도 토큰기반 인증 방식의 큰 장점 중 하나이다.
